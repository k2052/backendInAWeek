About your backend

Your backend will be in Ruby. It will be powered by Grape a minimalist web framework with a powerful DSL for expressing APIs. Tests will be written using RSpec. Test fixtures will be created using FactoryGirl. Any external API calls will be mocked using WebMock and or cached using VCR.

If background jobs are needed they will be written as Imperator commands and then run with either Resque or Sidekiq. JSON responses will be done using Oat. You will have your choice of a data store and ORM; MongoMapper, ToyStore, Mongoid, DataMapper or NoBrainer.

Your API will conform to json:api. It will be designed to be consumed as easily from the web as from mobile. It will be discoverable (think HATEOAS) and data will be as denormalized as possible. It will use tokens for authentication not cookies. It will be documented with API Blueprint and tested using Dredd.

I believe in separation of concerns, slim models and clean controllers. You wont find your backend polluted with logic in places where it doesn't belong. What you will find is a lot of interactors, value objects, commands, policies, and decorators. I believe in a readable codebase; if dogma gets in the way of that I throw it out quickly and go with what is more readable. I don't use a pattern when the codebase is cleaner without it. Patterns are just tools.

Process

API Blueprint

The process starts by documenting the API with an API Blueprint. Ideally, you will have already written this and provide it to me. For an extra fee I will write them for you.

Why do I start with API docs?

  1. It provides a guide mark for completion. How much of the API is passing is a good indicator of how much progress has been made. If something from the API spec is not built yet then the backend is not done. The project is not considered complete (no matter how much else is done) until the API is fully covered by tests and those tests are passing.
  2. You can build your app independently of its backend and then when its ready swap out the mocks for the real thing.
  3. It serves to scope the project and clearly lay out what we are agreeing on.
  4. Helps future developers and you know how everything works. Forgotten how a part of your backend works? Need to get a developer up to speed? Just consult the documentation.
